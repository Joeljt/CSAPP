## Solution

bomblab 提供了一个二进制文件 bomb，执行后提示输入内容，如果输出错误程序就会爆炸。

需要利用 gdb 进行 debug，反编译破解二进制文件，通过查看汇编指令还原程序逻辑，确定需要输入的内容是什么。

程序一共有 6 个阶段，难度系数按阶段增加；同时，还有一个隐藏的阶段，需要通过特殊方式进入。

### 第一阶段

执行 bomb 程序后设置断点 `b phase_1`，反编译该方法如下：

```assemby
(gdb) disas phase_1
Dump of assembler code for function phase_1:
   0x0000000000400ee0 <+0>:	sub    $0x8,%rsp
   0x0000000000400ee4 <+4>:	mov    $0x402400,%esi
   0x0000000000400ee9 <+9>:	callq  0x401338 <strings_not_equal>
   0x0000000000400eee <+14>:	test   %eax,%eax
   0x0000000000400ef0 <+16>:	je     0x400ef7 <phase_1+23>
   0x0000000000400ef2 <+18>:	callq  0x40143a <explode_bomb>
   0x0000000000400ef7 <+23>:	add    $0x8,%rsp
   0x0000000000400efb <+27>:	retq   
End of assembler dump.
```

通过分析代码可以发现：

1. 栈指针向下移动，开辟了 8 字节空间；
2. 更新寄存器 `%esi` 的值为 `0x402400`，这是一个内存地址值；
3. 之后调用 `strings_not_equal` 方法后，用 test 指令检查一致性；
4. 如果值相同就回收栈空间后返回，否则爆炸结束。

那问题就很清晰了，只需要确认地址 `0x402400` 存储的内容是什么即可：

```assembly
(gdb) x/s 0x402400
0x402400:	"Border relations with Canada have never been better."
```

第一个炸弹拆除完成。🎉

1. 第一个参数在寄存器 `%rdi` 中，表示输入的内容；
2. 第二个参数是格式化规格，储存在寄存器 `%rsi` 里，也就是 <+36> 的操作，检查后发现格式是 `%d %d %d %d %d %d`，意味着我们需要按照这个格式输入 6 个整数;

第三阶段的代码是 switch-case 的跳转表优化，从代码中可以看到：

1. 只能输入两个整数 `x` 和 `y`，按空格分隔；
2. 限制 `x` 只能是 [0, 7] 之间的某个数字；
3. 根据 `x` 的值进行间接跳转的地址值计算，计算规则是：`0x402470 + x * 8`，可以计算出来 8 个地址；
4. 可以发现有很多 `je 0x400fc9 <phase_3+134>` 条件跳转，对应的正是上面的 8 个地址；
5. 所以只需要按照跳转到的地址，确定 `x` 的值对应的 `y` 是哪个值即可；

第三阶段的重点在于跳转表的理解。

在 switch 语句的 case 值相对连续时，编译器会把 case 语句块单独封装到一个代码块里。

之后将所有的 case 语句块都放到一个数组里，在匹配的时候，直接用 case 值做数组索引，以 O(1) 的效率确定跳转到的语句块地址。

也就是说，这个跳转表数组是个指针数组，里面存放的是指向各个 case 代码块的起始位置的指针，即该起始位置的地址值，类似下面这样：

```c
[ 0x0003423234, 0x0003423873, 0x0003423256, 0x0003423852, ... ]
```

而汇编代码中间接跳转时候的 `0x402470` 就是这个跳转表中第一个元素所在的地址值，后面 8 个字节的偏移就是在偏移数组元素。

由于 C 语言中数组实际上就是指向第一个元素的指针，这个跳转表的声明大概就是：`int* *table`，表明这是个数组，里面存的元素都是地址值。

这也解释了为什么偏移量是 8 个字节，因为指针类型需要用 8 个字节存储。

另外还需要回顾的一点是，尽管 `(%rax, rbx, 5)` 经常被用来做算术运算，但是这个最原始的语义实际上是索引偏移寻址，最开始就是为数组索引元素设计的寻址方式，也就是我们遇到的这种场景。

综上，这道题目有 8 组可能的答案，比如：`0 207`、`1 707` 等。

第三个炸弹拆除完成。🎉

在观察各种寄存器操作的时候应该想到，机器执行一个指令只能做一件事，所以有可能很简单的一句代码翻译成指令需要两个甚至三个指令。

反过来说，很多看上去很复杂、很多、操作同一个寄存器的指令，有可能对应的是很简单的一句代码。

经过分析，我们可以发现这是一个递归函数，方法签名大概是：`int func4(int x, int start, int end)`。

其中，start 和 end 的初始值分别是 0 和 14，在方法内部对移位、取符号、合并等操作，直到运算结果与 x 相同时返回 0，否则就对运算结果 -1 后递归向下。

整个 `func4` 的执行逻辑大概是：

```c
int func4(int input, int start, int end) {
    // 二者相减加上结果的符号位做算术右移 + start
    int sign = (end - start) >> 31;
    int n = (end - start + sign) >>> 1 + start;

    if (input == n) return 0;

    if (input <= n) {
        // 如果计算的结果比输入值大，上界 -1 后递归
        return 2 * func4(input, start, --n);
    } else {
        // 如果计算的结果比输入值小，下界 +1 后递归
        int res = 0;
        res = func4(input, ++start, n);
        return 2 * res + 1;
    }
}
```

在这个问题中，虽然尝试还原了 C 代码，但是汇编代码因为有很多编辑器的优化导致逻辑注释并不好写清楚，还是需要 gdb 查看整个过程会更清晰。

在上层语言处理递归逻辑总有种晕晕的感觉，但实际上在汇编层，递归也只是普通的函数调用而已，根据条件进行跳转、调用、返回，所有的流程都和普通函数调用是一样的。

在这种情况下，我们就能更好得明白和理解：

- 递归过深会因为太多的函数调用导致栈帧过多，最终出现栈溢出问题；
- 函数的尾调用优化可以避免栈溢出，因为尾调用场景下根本不创建栈帧，所谓无论多深都不影响栈空间；
- 尾调用优化过的函数只是单纯的操作寄存器更新返回值，不只是不创建栈帧，甚至不访问内存，性能也更好；

具体在这个问题中，我们会发现程序会根据计算结果与输入值的大小对比，不停的调整上下界限，直到输入值与计算值完全相等才返回。

同时，通过对 return 进行运算操作，保证只有 input 的值是 0 的时候返回值才是 0，否则一定是正数。
